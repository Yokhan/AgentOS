# v0.2.33 Runtime Chat, Routing, and Code Context Plan

Date: 2026-04-26
Owner: AgentOS
Release target: v0.2.33

## Goal

Make AgentOS feel like a live orchestration cockpit without hiding the actual work:
chat must remain stable during long agent runs, cancellation must stop real provider work,
project routing must be visible, and plans must be saved as repo artifacts before execution.

## Non-goals

- Do not redesign the whole visual style in this pass.
- Do not mass-update external projects until AgentOS runtime is stable.
- Do not add a hard-coded "Codex only" workflow; provider and model routing must stay flexible.

## Phase 1: Saved Planning Contract

- Save this plan in `tasks/plans/` before code changes.
- Add release notes that point to the exact user-facing runtime fixes.
- Keep future execution plans as files, not only chat messages.

Acceptance:
- The release has a committed plan file.
- The plan lists verification and rollback criteria.

## Phase 2: Real Chat Cancellation

- Track the actual Codex CLI child process for chat runs.
- Let existing `stop_chat` kill the tracked process tree instead of only setting a flag.
- Prevent late provider output from being appended after cancellation.
- Preserve already streamed/run output when cancellation happens.

Acceptance:
- Starting a Codex chat run registers a PID under the chat key.
- Stopping the chat kills that PID tree on Windows.
- A cancelled run does not overwrite the visible transcript with stale output.

## Phase 3: Chat Durability and Runtime Feedback

- Keep current messages visible while a response is running or being cancelled.
- Avoid forced bottom-scroll unless the user is already near the bottom.
- Preserve at least the latest 200 messages and keep the door open for pagination.
- Surface backend/provider/runtime state in the chat area in a compact way.

Acceptance:
- User can scroll while the agent is thinking.
- Cancelling a long run does not erase already visible agent output.
- Chat reload errors do not clear the current transcript.

## Phase 4: Project Routing and Code Context

- Make the active route explicit: provider, model, project, and mode.
- Keep route controls provider-agnostic so future DeepSeek or other providers can fit.
- Prefer graph/code context as attached context, not another nested chat surface.
- Keep project-level delegation visualized as route/work items, not hidden commands.

Acceptance:
- The user can see which project/provider/model will handle the next message.
- The UI does not force a separate Duo panel for basic solo orchestration.
- Code graph context actions are discoverable from the work area.

## Phase 5: Verification and Release

- Run JS syntax checks for edited UI files.
- Run Rust checks/tests for edited backend paths.
- Run the existing npm test suite.
- Build updater artifacts.
- Bump version and release only after local verification passes.

Acceptance:
- `npm.cmd test` passes.
- `cargo test` or `cargo check` passes for backend changes.
- GitHub release workflow succeeds for `v0.2.33`.

## Rollback

- If process cancellation regresses provider runs, keep the UI changes but revert the chat-specific provider PID tracking.
- If updater build fails, do not tag the release.
- If GitHub release workflow fails, keep the commit but do not tell users to install until the run is green.
