# v0.3.20 Provider-Neutral Routing Plan

## Goal

Codex-only mode must work as a normal operating mode for AgentOS. Claude can be optional, disabled, or unavailable without breaking orchestration, delegation, backend APIs, strategy generation, or UI controls.

## Findings

- Several backend paths still called Claude directly: sync chat fallback, HTTP `/api/chat`, inbox processing, strategy generation, critical-signal auto-trigger, and delegation L3 permission escalation.
- Role model config was unsafe for fallback. With `claude_enabled=false`, old keys like `orchestrator_model=opus` could still be resolved for a Codex run.
- Settings UI implied Claude was the main orchestration/delegation runtime and did not expose `delegation_provider`.
- Provider status showed role providers but not the effective role model/effort, so the user could not verify what would actually run.

## Plan

1. Make provider resolution role-aware.
   - Claude roles use `orchestrator_model`, `orchestrator_effort`, `delegation_model`, and `delegation_effort`.
   - Codex roles use `orchestrator_codex_*` overrides if present, then `codex_model` and `codex_effort`.

2. Add one backend helper for orchestrator runs.
   - All non-streaming PA/orchestrator calls go through `run_orchestrator_once`.
   - The helper resolves provider, model, effort, permissions, and logs the effective route.

3. Replace direct Claude calls in execution paths.
   - Chat and HTTP API use `run_provider_with_opts`.
   - Strategy, inbox, auto-trigger, and delegation escalation use `run_orchestrator_once`.

4. Make Settings explain and control routing.
   - Add `delegation_provider` selector.
   - Show Claude fields as Claude-only.
   - Show Codex as a first-class route, not as an emergency fallback.

5. Add regression gates.
   - Test disabled-Claude routing with legacy Claude model values.
   - Run Rust lib tests and UI smoke checks before release.
