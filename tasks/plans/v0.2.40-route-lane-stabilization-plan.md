# Agent OS v0.2.40 Product Plan: Executable Orchestration Chat

## Methodology

This plan uses the `20 + 50` format as a product planning tool, not as a file-change checklist.

The 20 items are product capabilities: what becomes meaningfully better for the user and why it matters when managing many projects.

The 50 items are delivery slices: concrete backend, frontend, state, observability, and release changes that make the capabilities real and testable.

Each item must answer at least one of these questions:

- Does it reduce user uncertainty?
- Does it remove a manual orchestration step?
- Does it make agent work observable while it is happening?
- Does it make project routing safer across many projects?
- Does it preserve context instead of losing it in static UI or chat history?

## Product Thesis

Agent OS should feel like a live command center for multi-project agent work. The user should be able to discuss a situation, choose who leads, see what the system thinks the next executable route is, and start that route directly from chat.

The current weak point is not styling. The weak point is that the interface still sometimes behaves like a diagnostic dashboard: it shows state, but does not always turn state into an obvious safe action.

v0.2.40 focuses on one product invariant:

If Agent OS shows a route, it must either execute the next safe step or explain why execution is blocked.

## Before / After

Before:

- Route lanes looked actionable but often only inserted a prompt.
- Blocked delegations could disappear if no session/work-item lane existed.
- Old blockers could visually override active running work.
- Only the first four lanes were visible, hiding real project-agent state.
- The user had to infer whether a card meant "execute", "monitor", or "resolve".

After:

- Queueable lanes call the real work-item execution backend.
- Blocker-only routes appear as synthetic lanes.
- Running routes remain running and carry blocker metadata separately.
- Route board can expand to all returned lanes.
- Every route exposes an explicit action: queue next, monitor, resolve, or prompt.

## Primary User Scenarios

1. Solo orchestration: user talks to one selected model, sees route lanes, clicks the next safe project-agent route, and work starts.
2. Duo planning to execution: user discusses with two agents, selects a lead, then uses route lanes to execute work items produced by the plan.
3. Blocker triage: user sees that a project is not idle but blocked by a failed/rejected/permission delegation, and can route a resolution prompt.
4. Multi-project overview: user expands lanes to inspect more than four project-agent routes without switching screens.
5. Safe recovery: if a lane cannot queue work, the fallback prompt includes project, provider, reviewer, blocker ids, and suggested action.

## 20 Product Capabilities

1. Executable chat routing: route lanes become primary action controls, not decorative status cards.
2. Provider-neutral orchestration: routes describe executor/reviewer providers without assuming Codex-only flow.
3. Mode simplification: chat remains the universal surface; route actions do not require separate cockpit modes.
4. Safe next-step execution: queueable work items are started through the existing execution API, preserving permission checks.
5. Honest fallback behavior: non-executable lanes insert an explicit route prompt instead of silently pretending to execute.
6. Blocker visibility: failed, rejected, cancelled, and permission-blocked delegations stay visible even without normal lane state.
7. Active-work preservation: running/reviewing/lease activity stays visually dominant even when blockers exist.
8. Blocker metadata: active lanes expose `has_blockers` and `blocker_delegation_ids` so the user sees mixed state.
9. Expandable route board: default view stays compact, but all route lanes are accessible on demand.
10. Action semantics: each lane tells the user whether it can `queue next`, `monitor`, `resolve`, or `prompt`.
11. Reduced manual prompting: the common happy path removes the copy/edit/send loop.
12. Route refresh loop: execution actions refresh map and timeline so the UI reacts after work is queued.
13. Project load tolerance: route UI handles many active projects without hiding state behind a hard four-lane cutoff.
14. Graph-context continuity: code context remains visible as a routing input, not a separate dead-end tool.
15. Plan persistence: the product plan lives in repo state and can be audited after the chat context is gone.
16. Testable route semantics: backend tests cover ready, blocked-only, and running-with-blockers lanes.
17. Release confidence: updater build and CI release verify that product changes are installable.
18. Future provider support: route schema uses provider fields rather than hardcoded model assumptions.
19. User trust: visible state and actual behavior align, reducing "why did nothing happen?" moments.
20. Foundation for live operations: route lanes become the base for future streaming execution, reverse-channel reports, and per-project agent rooms.

## 50 Delivery Slices

### A. Route Execution UX

1. Add a route action decision: queue next if the backend marks the lane queueable.
2. Use `queueWorkItemExecution(next.id)` for queueable route lanes.
3. Keep prompt insertion only for lanes that cannot safely execute.
4. Show queue failures through the existing toast system.
5. Refresh orchestration map after queueing.
6. Refresh execution timeline after queueing.
7. Preserve current project and room-session context during refresh.
8. Add explicit fallback prompt fields: project, executor, reviewer, work item, blockers, suggested action.
9. Avoid hidden submenus for lane execution.
10. Keep lane click behavior consistent in solo and duo contexts.

## Acceptance Criteria

- A route lane with a ready agent work item queues that work item directly.
- A route lane without a queueable work item falls back to a useful prompt with enough context to continue.
- A failed/permission-blocked delegation appears even when there is no project session or active work item.
- Running work is not hidden by stale blockers.
- The user can inspect all returned project-agent routes from the chat card.
- Tests and updater build pass before release.

## Product Impact

This improves Agent OS as a product by converting state into action. The user no longer has to interpret a lane, write a prompt, and hope the agent chooses the right next step. The system now exposes a safe route, executes it when possible, and preserves blocker context when execution is not safe.

This is still not the final command-center UX. It is the foundation layer that makes the next wave possible: live route progress, per-project agent rooms, reverse-channel reports from project agents, and route recommendations across many projects.
