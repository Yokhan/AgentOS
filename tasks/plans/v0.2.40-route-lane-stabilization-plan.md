# Agent OS v0.2.40: продуктовый план исполняемого оркестрационного чата

## Методология

Формат `20 + 50` здесь используется не как список файлов и мелких правок, а как продуктовый план.

`20` - это крупные продуктовые улучшения: что станет заметно лучше для пользователя и почему это важно при ведении десятков проектов.

`50` - это поставочные срезы: конкретные UX, backend, state, observability, verification и release изменения, которые делают эти улучшения реальными и проверяемыми.

Каждый пункт должен отвечать хотя бы на один вопрос:

- Уменьшает ли это неопределенность пользователя?
- Убирает ли это ручной шаг в оркестрации?
- Делает ли это работу агентов видимой во время выполнения?
- Делает ли это маршрутизацию задач по проектам безопаснее?
- Сохраняет ли это контекст вместо потери в статичном UI или короткой истории чата?

## Продуктовая гипотеза

Agent OS должен ощущаться как живой командный центр для работы агентов по множеству проектов. Пользователь должен обсудить ситуацию, выбрать кто ведет, увидеть следующий безопасный маршрут выполнения и запустить его прямо из чата.

Слабое место сейчас не только в стиле. Слабое место в том, что интерфейс местами ведет себя как диагностический дашборд: он показывает состояние, но не всегда превращает это состояние в понятное безопасное действие.

Инвариант v0.2.40:

Если Agent OS показывает route, он должен либо выполнить следующий безопасный шаг, либо объяснить, почему выполнение заблокировано.

## Было / стало

Было:

- Route lanes выглядели как действия, но часто только вставляли текст в prompt.
- Заблокированные делегации могли пропасть, если под них не было session/work-item lane.
- Старые blockers могли визуально перекрыть реально выполняющуюся работу.
- UI показывал только первые четыре lanes, скрывая реальное состояние проектных агентов.
- Пользователь должен был сам угадывать, карточка означает execute, monitor или resolve.

Стало:

- Queueable lanes вызывают настоящий backend API выполнения work item.
- Blocker-only routes появляются как synthetic lanes.
- Running routes остаются running, а blockers показываются отдельной метаинформацией.
- Route board можно раскрыть и увидеть все возвращенные lanes.
- Каждая route lane явно показывает действие: queue next, monitor, resolve или prompt.

## Основные пользовательские сценарии

1. Solo orchestration: пользователь говорит с одной выбранной моделью, видит route lanes, кликает следующий безопасный маршрут проектного агента, работа стартует.
2. Duo planning to execution: пользователь обсуждает задачу с двумя агентами, выбирает lead, затем запускает work items из плана через route lanes.
3. Blocker triage: пользователь видит, что проект не idle, а заблокирован failed/rejected/permission delegation, и может направить prompt на разбор blocker.
4. Multi-project overview: пользователь раскрывает lanes и проверяет больше четырех проектных маршрутов без перехода на другой экран.
5. Safe recovery: если lane нельзя выполнить автоматически, fallback prompt содержит project, provider, reviewer, blocker ids и suggested action.

## 20 крупных продуктовых улучшений

1. Исполняемая маршрутизация из чата: route lanes становятся управляющими действиями, а не декоративными статус-карточками.
2. Provider-neutral orchestration: route описывает executor/reviewer provider без предположения, что всегда нужен Codex.
3. Упрощение режимов: чат остается универсальной поверхностью, route actions не требуют отдельного cockpit режима.
4. Безопасный next-step execution: queueable work items запускаются через существующий execution API с сохранением permission checks.
5. Честный fallback: non-executable lanes вставляют явный route prompt, а не делают вид, что выполнение запущено.
6. Видимость blockers: failed, rejected, cancelled и permission-blocked delegations не исчезают без normal lane state.
7. Сохранение active work: running/reviewing/lease activity остается визуально главным даже при наличии blockers.
8. Blocker metadata: active lanes показывают `has_blockers` и `blocker_delegation_ids`, чтобы mixed state был очевиден.
9. Expandable route board: дефолтный вид компактный, но все route lanes доступны по требованию.
10. Семантика действия: каждая lane говорит, что она может сделать: `queue next`, `monitor`, `resolve` или `prompt`.
11. Меньше ручного prompting: happy path убирает цикл “прочитал карточку -> написал prompt -> отправил -> надеюсь агент понял”.
12. Route refresh loop: после execution action обновляются map и timeline, чтобы UI реагировал на действие.
13. Устойчивость к большому числу проектов: route UI больше не прячет состояние за жестким лимитом в четыре карточки.
14. Непрерывность code context: graph/code context остается входом для route decision, а не отдельным тупиковым инструментом.
15. План сохраняется в репозитории: продуктовый план можно проверить после того, как чатовый контекст потерян.
16. Проверяемая семантика routes: backend tests покрывают ready, blocked-only и running-with-blockers lanes.
17. Release confidence: updater build и CI release подтверждают, что изменения реально устанавливаются.
18. Подготовка к новым providers: route schema использует provider fields, а не hardcoded модельные предположения.
19. Доверие пользователя: видимое состояние совпадает с поведением, меньше моментов “я нажал, а почему ничего не произошло?”.
20. База для live operations: route lanes становятся основой для live progress, reverse-channel reports и per-project agent rooms.

## Acceptance criteria

- Route lane с ready agent work item запускает этот work item напрямую.
- Route lane без queueable work item вставляет полезный prompt с достаточным контекстом.
- Failed/permission-blocked delegation видна даже без project session или active work item.
- Running work не скрывается stale blockers.
- Пользователь может увидеть все returned project-agent routes из chat card.
- Tests и updater build проходят до release.

## Продуктовый эффект

Это улучшает Agent OS как продукт, потому что превращает состояние в действие. Пользователю больше не нужно интерпретировать lane, писать prompt и надеяться, что агент выберет правильный следующий шаг. Система показывает безопасный route, выполняет его когда можно, и сохраняет blocker context, когда выполнение небезопасно.

Это еще не финальный command-center UX. Это фундаментальный слой для следующей волны: live route progress, per-project agent rooms, reverse-channel reports от проектных агентов и route recommendations по множеству проектов.
