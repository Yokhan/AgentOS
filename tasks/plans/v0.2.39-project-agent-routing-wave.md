# v0.2.39 Project-Agent Routing Wave

Date: 2026-04-26
Owner: AgentOS
Release target: v0.2.39

## Goal

Finish stage 6/6 of the UX/orchestration plan by making project-agent routing
visible as a first-class product surface: what project-agent lanes exist, what
they can pick up next, which provider owns execution/review, and what is blocked
by leases/delegations. Also harden the release workflow before GitHub's Node 24
action-runtime migration.

## Current Big-Plan Position

- Completed: foundation, Route Card, live transcript, orchestration map,
  execution timeline, shared event contract.
- Current: project-agent routing + release hardening.
- Next: normal product iteration from a complete 6/6 base.

## Top-Level Improvements

1. Add `project_agent_routes` to the backend orchestration map.
2. Build routes from project sessions, work items, delegations, and active leases.
3. Include executor/reviewer provider per route.
4. Include next work item per route.
5. Include route state: running, blocked, ready, idle.
6. Include active write lease counts.
7. Include failed/cancelled delegation counts as blockers.
8. Render route lanes in the Orchestration Map Card.
9. Add route-lane action prompts for handing off the next task.
10. Move big-plan metadata to stage 6/6.
11. Add backend tests for route lane state.
12. Opt the release workflow into Node 24 action runtime.
13. Bump version to `0.2.39`.
14. Update release notes.
15. Verify JS/Rust/build/release assets.

## Acceptance Criteria

- The chat UI shows concrete project-agent lanes, not only raw project sessions.
- A user can tell which lane is running, ready, blocked, or idle.
- A user can see the next work item and responsible provider for each lane.
- The product itself says the big plan is at stage 6/6.
- GitHub release workflow no longer relies on Node 20 action runtime.
- `v0.2.39` updater assets publish successfully.
