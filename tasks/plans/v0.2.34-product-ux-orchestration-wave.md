# v0.2.34 Product UX and Orchestration Wave

Date: 2026-04-26
Owner: AgentOS
Release target: v0.2.34

## Goal

Turn AgentOS chat from a static command panel into a responsive orchestration surface:
one chat is the primary interface, route state is always visible, project/delegation
work is understandable, and code graph context can be attached to model work without
forcing a nested panel.

## Design Rules

- Chat is the primary interface. Tools, plans, graph, delegations, and providers report into chat.
- Solo mode must be enough to tell any selected provider what to do.
- Duo is a review/execution room, not a second chat inside the first chat.
- Provider routing must stay generic: Claude, Codex, future DeepSeek, and ACP adapters all fit.
- Controls should be progressive: common actions visible, rare routes behind compact details.
- Runtime state must feel live: work should move, stream, expose blockers, and preserve output.

## 20 Top-Level Improvements

1. Saved execution plans: every large wave gets a committed plan file before code.
2. Route model: unify target project, provider, model, mode, access, and lead into one route object.
3. Route visibility: show the next-message route in one compact, readable surface.
4. Route actions: make common pivots one-click: plan, act, review, add context, execute lead.
5. Provider neutrality: remove UX assumptions that execution means only Codex.
6. Model freshness: show whether provider models came from runtime/cache/fallback.
7. Runtime HUD: show active run state, elapsed time, cancellation state, and latest event.
8. Stop confidence: show whether Stop killed a process or only requested cancellation.
9. Chat history durability: make older context available without loading entire JSONL files.
10. Scroll stability: never force-scroll while the user is reading older output.
11. Cancellation durability: already-rendered output must stay visible after Stop.
12. Project routing: make selected project, global orchestrator, and graph scope obvious.
13. Delegation map: show active/pending/dead delegations as work, not hidden command text.
14. Project agent handoff: show which project agent/session a task would enter.
15. Duo synthesis flow: discuss with two agents, convert to plan, choose lead, execute.
16. Project-level plans: distinguish global strategy, project plan, and immediate task route.
17. Graph/code context: expose context/impact/dependents/verify from graph and chat.
18. Command trace UX: render PA command batches as structured run cards, not stdout walls.
19. Backend observability: expose enough metadata for UI to explain what is happening.
20. Release discipline: test, build updater, tag, publish, and verify release assets.

## Acceptance Criteria

- A user can stay in solo mode and tell the selected provider to plan or execute.
- A user can discuss with Duo, convert the discussion to a plan, choose a lead, and execute without guessing where input goes.
- The route surface explains target, provider, model, mode, access, runtime state, and blockers.
- The chat does not lose output on cancellation or history load.
- Code graph context can be attached or sent from the graph inspector.
- Release assets and updater manifest are published for `v0.2.34`.

## Rollback

- If the route card crowds the composer, keep backend metadata and revert route-card rendering.
- If graph attach actions break immediate graph prompts, keep "ask now" and hide "attach" until fixed.
- If release build fails, do not tag.
- If GitHub release workflow fails, keep commit but do not tell users to install.
