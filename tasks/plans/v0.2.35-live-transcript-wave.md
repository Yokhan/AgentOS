# v0.2.35 Live Transcript Wave

Date: 2026-04-26
Owner: AgentOS
Release target: v0.2.35

## Goal

Make chat behave like a live work console instead of a static log: the user can
read older output without being pulled down, see whether the transcript is
following or paused, keep partial output after stop/failure, and understand what
new work arrived while reading.

## Design Rules

- Chat owns the work loop. Runtime state, command output, history, and route
  state should surface in chat, not in hidden side channels.
- Auto-scroll must be conditional. Follow live output only while the user is at
  the bottom or explicitly requests it.
- Already-rendered output is user data. Do not erase it merely because backend
  persistence, cancellation, or refresh timing is awkward.
- Remove hidden duplicate controls. If a route/control is obsolete, delete it
  from DOM instead of hiding it with CSS.
- Keep the implementation small and reversible: no new framework, no invasive
  stream backend rewrite in this wave.

## Top-Level Improvements

1. Add a transcript status bar above messages.
2. Track whether chat is following live output or the user is reading history.
3. Count new live updates while reading older output.
4. Provide one-click "follow latest" without forcing scroll during reading.
5. Keep partial live output visible if final JSONL persistence is missing.
6. Mark unpersisted live output as an explicit recovered transcript, not a ghost.
7. Make Stop settle through visible state instead of clearing the stream bubble.
8. Remove old hidden solo/duo route strips from the DOM.
9. Replace inline scroll button styling with stylesheet-backed UI.
10. Add release notes and tests/build/release discipline for the wave.

## Concrete Changes

2. Add local "following/reading" viewport state derived from scroll position.
3. Add `TranscriptStatusBar` component.
4. Show loaded/total history in the transcript bar.
5. Show active run phase/detail in the transcript bar.
6. Show live update count while user is reading above the bottom.
7. Add "follow latest" button to the transcript bar.
8. Add "load older" action to the transcript bar when history exists.
9. Keep the existing top `load older` button as a fallback while the bar matures.
10. Replace the floating arrow glyph with an ASCII `latest` button.
11. Move floating scroll button styling into `chat.css`.
12. Increment unread count on new side messages while reading.
13. Increment unread count on stream text/chain updates while reading.
14. Reset unread count when user scrolls near bottom or clicks follow.
15. Preserve stream text/chain if final chat reload cannot find persisted output.
16. Label preserved stream output as `recovered live output`.
17. Keep preserved output for cancelled and failed runs.
18. Keep preserved output for done runs if JSONL persistence is not detected.
19. Add a toast when live output is preserved because persistence was not found.
20. Remove hidden legacy `.solo-compose-route` JSX block.
21. Remove hidden legacy `.duo-compose-route` JSX block.
22. Remove CSS rules whose only job was hiding those old blocks.
23. Bump app version to `0.2.35`.
24. Add user-facing release notes.
25. Verify with JS syntax checks, cargo tests, updater build, tag, and release.

## Acceptance Criteria

- Scrolling up during an active run does not snap back to bottom.
- New output while reading is visible as a count and can be followed explicitly.
- Stopping a run does not erase already-rendered output.
- If backend history reload misses the assistant output, the live transcript
  remains visible and clearly labeled.
- The old hidden route strips are gone from DOM.
- `v0.2.35` updater assets publish with `latest.json`.
