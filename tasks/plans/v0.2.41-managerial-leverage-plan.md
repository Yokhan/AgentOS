# Agent OS v0.2.41: продуктовый план managerial leverage

## Цель

Agent OS должен повышать эффективность пользователя как управляющего нейросетями: увеличивать параллельную работу, повышать качество за счет перекрестных проверок между разными агентами, снижать глубину ручного контроля и держать execution привязанным к целям, стратегиям и планам.

## Продуктовая гипотеза

Пользователь не должен вручную помнить, какие агенты что делают, где нужен reviewer, какие задачи оторваны от стратегии, и где система ждет человека. Agent OS должен показывать это как live management layer прямо в чате.

Главный инвариант:

Каждый видимый work route должен отвечать на четыре вопроса: зачем это делается, кто выполняет, кто проверяет, где нужен контроль пользователя.

## 20 крупных продуктовых улучшений

1. Managerial leverage score: единый индикатор, насколько система реально разгружает пользователя.
2. Parallel capacity: видимость активных проектных lanes, running work, queueable routes и проектов в движении.
3. Quality coverage: явная метрика, какая доля работы имеет reviewer/cross-check.
4. Control load: подсчет мест, где требуется пользовательское вмешательство.
5. Strategy alignment: понимание, привязана ли текущая работа к активной стратегии/планам.
6. Goal drift detection: сигнал, когда work items существуют без strategy/plan source.
7. Cross-agent quality: отдельная видимость executor/reviewer pair, особенно когда это разные providers.
8. Blocker pressure: blockers учитываются не как шум, а как контрольная нагрузка.
9. Queue readiness: route board показывает не только состояние, но и объем стартуемой работы.
10. User attention budget: UI явно говорит, сколько decisions/blockers висит на пользователе.
11. Management recommendations: система сама формулирует следующий управленческий шаг.
12. Reduced micromanagement: пользователь видит агрегированную картину вместо чтения каждого лога.
13. Better delegation trust: cross-check coverage снижает необходимость вручную читать все ответы.
14. Project-level routing: метрики работают как в global scope, так и внутри проекта.
15. Room-level routing: метрики сохраняют связь с текущей Duo/session комнатой.
16. Code context readiness: graph context включен в оценку безопасного execution.
17. Operational observability: map показывает не только entities, но и состояние управляемости.
18. Testable management semantics: вычисления leverage покрываются backend tests.
19. Release-safe rollout: изменения идут через updater build и GitHub release.
20. Foundation for future agent board: этот слой станет базой для per-project agent rooms и live cross-review.

## Acceptance criteria

- В chat orchestration map виден management layer, а не только route/debug state.
- Пользователь видит, насколько система параллелит работу, где качество покрыто reviewer, и где нужен его контроль.
- Работа без стратегии/плана подсвечивается как alignment risk.
- UI предлагает следующий управленческий шаг, а не только показывает счетчики.
- Backend tests подтверждают leverage, quality и control-load semantics.
